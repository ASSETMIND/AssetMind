## 1. 개요 (Overview)

본 문서는 AssetMind Stock 시스템의 차트 데이터 조회 API에 대한 **순수 성능 한계치(Baseline) 진단 및 병목 구간 분석 명세**이다.
k6 부하 테스트 도구를 활용하여 클라이언트의 과거 차트 무한 스크롤(동적 페이징) 시나리오를 재현하였으며, 고도화 이전에 시스템이 버틸 수 있는 최대 임계점과 아키텍처 전반(DB, App, Infra, Ext)의 취약점을 문서화하여 다음 스프린트의 최적화 지표로 삼는다.

### 1.1. 측정 대상 및 진단 목표
- **부하 시나리오**: `GET /api/stocks/{stockCode}/charts/candles` 동적 페이징 무한 호출
- **진단 목표**: 가상 유저 500명 도달 시 서버의 Avg TPS, p(95) Latency 한계치 측정
- **병목 색출**: 트래픽 포화 상태에서 발생하는 DB I/O 지연, 컨테이너 리소스 경합, 외부 API 세션(KIS) 연쇄 장애 규명

## 2. 부하 테스트 한계치 측정 결과 (Baseline)

가상 유저(VUs) 500명을 투입한 극한 부하 테스트 결과, 톰캣 스레드 및 DB 커넥션 병목으로 인해 심각한 지연과 에러가 발생함을 확인하였다.

| 측정 지표 | 결과 수치 | 상태 | 비고 (분석) |
| :--- | :--- | :---: | :--- |
| **평균 처리량 (Avg TPS)** | **7.43 req/s** | 🚨 | 요청 적체로 인해 500명의 부하 대비 초당 처리 능력이 극도로 저하됨 |
| **p(95) 지연 시간** | **52.32 s** | 🚨 | 하위 5%의 사용자는 차트 조회에 최대 52초 이상 대기 (Timeout 임계치 도달) |
| **에러율 (Error Rate)** | **5.52 %** | 🚨 | 총 3,784건 중 209건이 500 Internal Server Error 또는 Timeout 처리됨 |
| **최소 응답 속도 (Min)** | **17.42 ms** | ✅ | 부하 인가 극초반, 병목이 발생하기 전의 정상 응답 속도 (목표치) |

## 3. 계층별 성능 병목(Bottleneck) 진단 결과

성능 저하를 유발한 4가지 핵심 병목 구간을 아키텍처 계층별로 분류하여 분석하였다.

| ID | 병목 분류 (Layer)                 | 원인 분석 (Root Cause) | 현상 및 증명 (Symptom) | 해결 방안 (Action Item) |
| :--- |:------------------------------| :--- | :--- | :--- |
| **BOT-DB-001** | **[TCS-DB]**<br>DB I/O 과부하    | 대용량 시계열 테이블(1,000만 건)에 대한 인덱스 부재 | 쿼리 지연으로 인한 HikariCP 커넥션 풀 고갈 및 대기열 생성 | `stock_code` + `candle_timestamp` 복합 인덱스(Composite Index) 추가 |
| **BOT-APP-001** | **[TCS-APP]**<br>연산 오버헤드     | `date_bin` 그룹화 쿼리 실행 시 `WHERE` 절에 하한선(`startTime`) 조건 누락 | 2000년도부터의 과거 캔들을 모두 메모리에 올려 정렬(`array_agg`)하는 불필요한 Full 탐색 발생 | Service 계층에서 요청 `limit` 기반으로 넉넉한 `startTime` 버퍼를 계산하여 쿼리에 주입 |
| **BOT-INF-001** | **[TCS-INFRA]**<br>리소스 격리 실패 | DB 연산 폭증 시 PostgreSQL 컨테이너가 Docker(Colima)의 CPU를 100% 독점 (Noisy Neighbor) | 동일 환경 내의 Prometheus 컨테이너가 CPU를 할당받지 못해 지표 수집 타임아웃(`No data`) 발생 | `docker-compose.yml` 내 각 컨테이너별 Cgroups CPU/Memory 제한(`Limits`) 명시적 설정 |
| **BOT-EXT-001** | **[TCS-EXT]**<br>연쇄 장애(SPOF) | 부하로 인한 서버 CPU 임계치 도달 및 잦은 GC(Stop-The-World) 발생 | KIS 실시간 웹소켓 세션이 Keep-alive(Ping/Pong) 처리를 하지 못해 외부 연결이 강제 종료됨 | 1. 웹소켓 지수 백오프(Exponential Backoff) 재연결 로직 적용<br>2. API 서빙과 데이터 수집(Worker) 서버 분리 아키텍처 검토 |
