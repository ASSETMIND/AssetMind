---

## 1. 개요
본 문서는 AssetMind 시스템이 제공하는 기능에 대해 시스템과 사용자(또는 외부 시스템) 간의 상호작용을 명세하는 '유즈케이스(Use Case)' 작성 표준을 정의한다. 유저 스토리가 '무엇을, 왜'에 집중한다면, 유즈케이스는 **'어떻게(흐름)'**에 집중하여 개발 및 테스트의 구체적인 설계도가 된다.

---

## 2. 유즈케이스 구성 요소
하나의 유즈케이스는 시스템의 구체적인 행동 흐름을 파악할 수 있도록 다음 요소들을 포함해야 한다.

| 항목 | 설명 | 비고 |
| :--- | :--- | :--- |
| **Use Case ID** | `UC-[주제약어]-[번호]` 형식 | 예: `UC-SYS-001` |
| **유즈케이스명** | 기능을 명확히 표현하는 '동사+명사' 형태 | 예: 이메일 로그인 |
| **관련 액터 (Actor)** | 해당 기능을 수행하는 주체 (사용자, 관리자, 외부 시스템 등) | |
| **사전 조건 (Preconditions)** | 기능 수행 전 반드시 만족해야 하는 시스템 상태 | |
| **종료 조건 (Postconditions)** | 기능 수행 완료 후 변경되는 시스템 상태 | |
| **기본 흐름 (Basic Flow)** | 오류 없이 성공적으로 수행되는 시나리오 (Step-by-step) | Happy Path |
| **대안/예외 흐름 (Alt/Exc Flow)** | 조건 불만족, 오류 발생 시 분기되는 시나리오 | |

---

## 3. 상세 작성 규칙

### 3.1. 흐름(Flow) 작성 원칙
* **주어 명시**: '누가' 행위를 하는지 명확히 한다. (예: 사용자는..., 시스템은...)
* **단문 사용**: 한 단계에는 하나의 행위만 기술한다.
* **시스템 응답 포함**: 사용자의 요청에 대해 시스템이 어떻게 반응(검증, 계산, 저장, 출력)하는지 반드시 기술한다.

### 3.2. 유저 스토리와의 관계
* 일반적으로 1개의 유저 스토리는 1개 이상의 유즈케이스로 구체화된다.
* 유즈케이스 명세서는 **REQ(요구사항)**와 **TCS(테스트케이스)**를 연결하는 다리 역할을 한다.

---

## 4. 작성 예시

### 예시: 서비스 로그인 (UC-SYS)

| 항목 | 내용 |
| :--- | :--- |
| **Use Case ID** | `UC-AUTH-001` |
| **유즈케이스명** | 이메일/비밀번호 기반 로그인 |
| **개요** | 가입된 사용자가 자신의 계정 정보(이메일, 비밀번호)를 입력하여 시스템 접근 권한을 획득한다. |
| **관련 액터** | 사용자 (비로그인 상태) |
| **사전 조건** | 1. 사용자가 회원가입을 완료하여 활성 상태의 계정을 보유하고 있어야 한다.<br>2. 사용자가 로그인 페이지에 접속한 상태여야 한다. |
| **종료 조건** | 1. **(성공)** 인증 토큰(JWT)이 발급되고 브라우저(로컬 스토리지/쿠키)에 저장된다.<br>2. **(성공)** 사용자는 메인 대시보드 화면으로 이동한다.<br>3. **(실패)** 로그인 실패 횟수가 증가하고, 에러 메시지가 표시된다. |
| **기본 흐름**<br>**(Basic Flow)** | 1. **[사용자]** 이메일과 비밀번호를 입력하고 '로그인' 버튼을 클릭한다.<br>2. **[시스템]** 입력된 이메일 형식(Regex)과 비밀번호 입력 여부를 클라이언트 측에서 1차 검증한다.<br>3. **[시스템]** 입력받은 정보를 암호화하여 서버로 전송한다.<br>4. **[시스템]** DB에서 해당 이메일의 계정 존재 여부와 비밀번호 일치 여부를 확인한다.<br>5. **[시스템]** 인증 성공 시, Access Token과 Refresh Token을 생성하여 반환한다.<br>6. **[시스템]** 최근 접속 일시를 현재 시간으로 갱신(Update)한다.<br>7. **[시스템]** 메인 대시보드 페이지로 리다이렉트한다. |
| **대안/예외 흐름**<br>**(Alternative Flow)** | **2a. 입력 형식 오류**<br>1. 사용자가 이메일 형식이 아닌 값을 입력하거나 비밀번호를 비워둔 경우.<br>2. 시스템은 입력 필드 하단에 "올바른 이메일 형식을 입력해주세요" 또는 "비밀번호를 입력해주세요" 메시지를 출력하고 전송을 차단한다.<br><br>**4a. 인증 실패 (정보 불일치)**<br>1. 등록되지 않은 이메일이거나 비밀번호가 틀린 경우.<br>2. 시스템은 "아이디 또는 비밀번호가 일치하지 않습니다." 메시지를 출력한다.<br>3. 시스템은 해당 계정의 로그인 실패 횟수를 1 증가시킨다.<br><br>**4b. 계정 잠금 (5회 이상 실패)**<br>1. 로그인 실패 횟수가 5회에 도달한 경우.<br>2. 시스템은 계정을 '잠금(Locked)' 상태로 변경한다.<br>3. 시스템은 "보안을 위해 계정이 잠겼습니다. 비밀번호 찾기를 진행해주세요." 메시지를 출력한다. |
| **관련 문서** | `US-SYS-001`, `API-AUTH-001`, `DB-USER-01` |
